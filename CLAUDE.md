# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A Tkinter desktop "Paint" app built for a Computer Graphics (GPC) university course. Its entire purpose is educational: **no native Tkinter drawing functions are used for the primitives themselves** — every line/circle/ellipse/parabola pixel is computed by a pure-Python algorithm class (DDA, Bresenham, Punto Medio) and then plotted one pixel at a time onto the canvas.

## Commands

- Run the app: `python3 main.py`
- No third-party dependencies — stdlib + Tkinter only (no `requirements.txt`/`pyproject.toml`, no venv needed beyond a standard Python 3 install with Tk support).
- No test suite, linter, or formatter is configured in this repo. Verify changes by running the app and exercising the feature manually through the GUI (this is a GUI-driven app with no automated tests).
- Each module in `algorithms/` has an `if __name__ == "__main__":` demo block that prints computed points to stdout — useful for sanity-checking an algorithm in isolation without launching the GUI, e.g. `python3 -m algorithms.elipse` or `python3 algorithms/circunferencias.py`.

## Architecture

Simplified MVC, split across three packages:

- `algorithms/` — pure computation, zero Tkinter imports. Each class takes geometric parameters in its constructor and exposes a `generate_points()`/`generate_initial_points()` method returning plain `(x, y)` integer tuples. `Circle` (in `circunferencias.py`, DDA variant is `CircleDDA`) and `ElipsePM` (in `elipse.py`) additionally expose `generate_octant_points()` / `generate_quadrant_points()`, which take the first-octant/quadrant points and mirror them by symmetry into a dict keyed `octant_1..8` / `quadrant_1..4`. These classes are never touched by Tkinter — they're plotted by the controller. `algorithms/__init__.py` re-exports every algorithm class, so the controller imports from the package (`from algorithms import DDA, CircleDDA, PuntoMedio, Bresenham, ElipsePM, ParabolaPM`) rather than each submodule individually.
- `ui/main_window.py` — the view (`MainWindow(tk.Tk)`). All layout is built procedurally in one method, `_create_layout()`. It has no logic of its own: every widget it creates is exposed as a public attribute (`self.btn_dda`, `self.color_squares`, `self.canvas_manager`, `self.log_text`, etc.) so the controller can wire up commands/bindings after construction.
- `ui/canvas_manager.py` — thin renderer around the `tk.Canvas`. `draw_pixel(x, y, color, size)` is the one primitive everything funnels through (a Tkinter rectangle simulating a pixel — this is the only "native" drawing call in the app, used purely as the pixel-plotting unit, not as a shape primitive). `redraw(strokes, grid_enabled, dark_mode)` clears the canvas and repaints in layer order: background color → grid lines → every stroke in the history. `CanvasManager` holds no drawing history itself — it's a stateless renderer driven by data the controller passes in.
- `controllers/app_controller.py` — all business logic and event wiring lives here (`AppController`). Key things to know before touching it:
  - **Tool/mode selection uses string keys**, not an enum: `self.current_mode` is one of the literal strings used as keys in `self.tool_buttons` (e.g. `"LINEA DDA"`, `"CIRCULO PM"`, `"ELIPSE PM"`). Adding a new primitive means adding a new string mode, not a new type.
  - **Two input paths**: mouse clicks (`on_canvas_click`) drive lines via two sequential clicks (start point, then end point). Circles/ellipses (and any primitive needing more than 2 raw coordinates) are *not* mouse-driven — clicking the canvas while one of those modes is active just tells the user to use the "Coordenadas" button, which opens a modal (`open_coordinates_popup`) with fields built dynamically based on `self.current_mode`.
  - **Stroke history**: `self.strokes` is a list of dicts (`{"tool", "points", "color", "thickness", "groups"}`) appended to whenever a shape finishes drawing (see `_record_stroke`); `self.redo_stack` holds undone strokes. `"groups"` is a list parallel to `"points"` giving each point's group label (e.g. `"Octante 3"`, `"Cuadrante 1"`, `"Rama 2"`, or `"N/A"` for lines/tools with no symmetry grouping) — `_draw_circle_and_ellipse_detail` derives it by enumerating `points_dict` (dict insertion order = group order), so it works for any primitive with a symmetry dict without primitive-specific branching. This is what `canvas_manager.redraw()` replays (points/color/thickness only — `groups` is metadata, not used for rendering), and it's the backing data for undo/redo, CSV export, and any grid/dark-mode toggle (which must call `redraw()` to reapply strokes on top of the new background).
  - **Widget wiring pattern**: the controller's `__init__` reaches into `self.view.<widget_name>` to call `.config(command=...)` or `.bind("<Button-1>", ...)`. New widgets should be defined in `MainWindow._create_layout`, exposed as `self.<name>`, and wired here — never build widgets inside the controller.
  - **Two visual idioms for buttons**: `ttk.Button` is used for tool-selection and modal-opening actions (line/circle/ellipse buttons, "Coordenadas", "⚙️ Opciones"); plain `tk.Label` with a `<Button-1>` bind and inline hex colors (mimicking Bootstrap primary/danger buttons) is used for destructive/state actions like "Limpiar Pantalla" and "Salir". Follow the existing idiom for the kind of action you're adding rather than mixing them.
  - The log panel (`self.view.log_text`, a `ScrolledText`) doubles as a place to inject live widgets — `_draw_circle_and_ellipse_detail` uses `log_text.window_create()` to embed a `ttk.Button` inline in the log that opens a detail window (`show_points_details`) listing all mirrored points.

## Current status & roadmap

All 7 originally planned features are done, built in dependency order (not the order they were originally requested — see rationale below); Fase 7 (CSV import) was added afterward as a counterpart to Fase 2's export. Full design detail for Fases 0–5 lives in the plan file `~/.claude/plans/tengo-una-lista-de-logical-eclipse.md`; Fases 6–7 were planned in `~/.claude/plans/lee-el-claude-md-y-wondrous-crane.md`. This section is the quick-reference checklist of what's done.

**Why this order**: none of Undo/Redo, Grid, Dark Mode, CSV Export, or Animation could be built in isolation, because the app had no persistent record of what had been drawn (`draw_pixel` painted directly onto the canvas and forgot the point immediately). Fase 0 introduces that record (`self.strokes` + `CanvasManager.redraw()`) and everything else builds on top of it. Undo/Redo was also moved ahead of Animation, because Animation needs to add "don't redraw mid-animation" guards into code that Undo/Redo and the Grid/Dark-Mode toggles create — so those need to exist first.

- [x] **Fase 0 — Fundación (historial de trazos + `redraw()`)** — done. `CanvasManager.draw_pixel` takes a `size` param, `draw_grid()` and `redraw()` exist. `AppController` has `self.strokes`, `self.redo_stack`, `self.grid_enabled`, `self.dark_mode`, `self.current_thickness`, and a `_record_stroke()` helper that `draw_lines` / `_draw_circle_and_ellipse_detail` call after painting. `clear_canvas()` now resets both stacks and goes through `redraw()`.
- [x] **Fase 1 — Algoritmo de Parábola** — done, and mouse-driven (deviation from the original plan, which had it Coordenadas-only like circle/ellipse). `algorithms/parabola.py` (`ParabolaPM(x1, y1, x2, y2)`) takes a start point + end point (same field names as the line tool) and rasterizes with a real 2-region midpoint algorithm (region 1 dominant-Y near the vertex, region 2 dominant-X further out — without both regions a large scale factor degenerates into a straight line, verified during implementation). `"PARABOLA PM"` is wired into `tool_buttons`, `on_canvas_click` (2-click mouse flow like lines, dispatching to `draw_parabola` instead of `draw_lines`), and `open_coordinates_popup` (reuses the line tool's exact field set: `X Inicial/Y Inicial/X Final/Y Final`). `_draw_circle_and_ellipse_detail`/`show_points_details` were generalized from a circle/ellipse ternary to a `PRIMITIVE_INFO` dict keyed by primitive name (`circle`/`ellipse`/`parabola`).
- [x] **Fase 2 — Exportación de Coordenadas (CSV)** — done. `utils/export_data.py` (`ExportData`) takes the stroke list, a filename, and a column list, writing one row per point (`trazo_id, herramienta, x, y, color, grosor, grupo`) — column→value mapping lives in a `field_getters` dict keyed by column name, so `export_to_csv(columns)` only writes whichever columns are requested. `grupo` reads the per-point `stroke["groups"][i]` (see Stroke history note above), giving values like `"Octante 3"`/`"Cuadrante 1"`/`"Rama 2"`/`"N/A"`. `AppController.export_to_csv()` prompts via `filedialog.asksaveasfilename` and delegates to it. New `self.btn_export` (`ttk.Button`) pinned to the right side of the top bar (`pack(side=tk.RIGHT)`, separate frame from the left-aligned "Coordenadas"/"Entrada" controls).
- [x] **Fase 3 — Grosor de Trazo** — done. `ui/main_window.py` adds `self.thickness_var` (a `tk.IntVar`, default 1) and `self.spinbox_thickness` (a `ttk.Spinbox`, range 1–10) to the top bar, after the color squares. `AppController.__init__` wires `self.view.thickness_var.trace_add("write", self._on_thickness_change)`; `_on_thickness_change` reads the var (guarding `tk.TclError` for the empty/in-progress-edit state while typing) and updates `self.current_thickness`, which `draw_lines`/`_draw_circle_and_ellipse_detail` already read when calling `draw_pixel` and `_record_stroke` — so the change applies only to new strokes, not retroactively.
- [x] **Fase 4 — Deshacer / Rehacer** — done. `AppController.undo_last_stroke()` / `redo_last_stroke()` pop between `self.strokes` and `self.redo_stack`, each followed by `canvas_manager.redraw(self.strokes, self.grid_enabled, self.dark_mode)`; an empty stack just logs a warning instead of erroring. `_record_stroke()` already clears `redo_stack` on every new stroke (see Stroke history note above), so redo is correctly invalidated once the user draws again after an undo. UI: `self.btn_undo`/`self.btn_redo` (`↩️`/`↪️`) are gray (`#6c757d`) `tk.Label`s — same Bootstrap-style idiom as `btn_clear`/`btn_exit`, but placed in the top bar's `mode_frame` (far left, before the "Entrada:"/"Coordenadas" controls) rather than the left tools panel, since the user wanted them reachable from the top menu.
- [x] **Fase 5 — Cuadrícula de Fondo + Modo Oscuro del Lienzo** — done. `open_options_popup` gained two `Checkbutton`s ("Mostrar cuadrícula", "Modo oscuro del lienzo") bound to `tk.BooleanVar`s seeded from `self.grid_enabled`/`self.dark_mode`; each has its own `command` callback (`toggle_grid`/`toggle_dark_mode`) that flips the corresponding controller flag and immediately calls `canvas_manager.redraw(self.strokes, self.grid_enabled, self.dark_mode)` — no need to hit "Aplicar y Cerrar" first, unlike the font-size control in the same popup. `CanvasManager.draw_grid()`/`redraw()` needed no changes; they already supported both flags since Fase 0. Dark mode only changes canvas `bg` and grid line color — the rest of the app's UI stays unaffected.
- [x] **Fase 6 — Animación de Trazado (renderizado progresivo)** — done. `AppController` gained `self.animation_enabled` (default `False`), `self.animation_delay` (ms between points, default `20`), and `self.is_animating`. New `_animate_points(points, on_point=None, on_complete=None)` helper recursively schedules one `draw_pixel` per point via `self.view.after(self.animation_delay, ...)`; `draw_lines` and `_draw_circle_and_ellipse_detail` (shared by circle/ellipse/parabola) branch on `self.animation_enabled` — animated path delegates to `_animate_points` with a `finish()` closure doing the `_record_stroke`/log/detail-button work that used to run inline, synchronous path keeps the original `for` loop followed by the same `finish()`. While `self.is_animating` is `True`, `draw_lines`, `_draw_circle_and_ellipse_detail`, `undo_last_stroke`, `redo_last_stroke`, `clear_canvas`, `toggle_grid`, `toggle_dark_mode`, and `import_from_csv` all bail out early with a log warning instead of mutating `self.strokes`/calling `redraw()`, since that would race with pixels the animation is still painting for a stroke not yet recorded. UI: `open_options_popup` gained a third section — "Animar trazado" `Checkbutton` (immediate `command`, like the Fase 5 grid/dark toggles) plus a "Velocidad (ms/punto)" `ttk.Spinbox` bound via `trace_add("write", ...)` (same `tk.TclError`-guard pattern as `_on_thickness_change`) — both apply live, no "Aplicar y Cerrar" needed.
- [x] **Fase 7 — Importar CSV** — done, symmetric counterpart to Fase 2. New `utils/import_data.py` (`ImportData`) reads a CSV in the same shape `ExportData` writes, groups rows by `trazo_id` (preserving file order), and reconstructs stroke dicts directly from the stored `(x, y)` pixels — it does **not** re-run any `algorithms/` class, since the CSV never stores original constructor params (center/radius/endpoints), only the already-rasterized pixel trail; this mirrors how `canvas_manager.redraw()` already replays strokes from `points`/`color`/`thickness` alone. Missing optional columns fall back to sane defaults (`color="black"`, `thickness=1`, `group="N/A"`, `tool="IMPORTADO"`); missing `x`/`y` raises, caught in the controller. `AppController.import_from_csv()` prompts via `filedialog.askopenfilename`; if the canvas already has strokes it calls `_ask_import_mode()` — a custom 3-button `Toplevel` ("Reemplazar"/"Añadir"/"Cancelar", using `wait_window` to block until closed, chosen over `messagebox.askyesnocancel` to control the Spanish button labels) — otherwise it imports straight in. New `self.btn_import` (`ttk.Button`) sits left of `self.btn_export` in the top bar's right-aligned group.
